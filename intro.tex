\begin{center}
\textbf{\Large Адаптивное планирование запросов в реляционных СУБД}
\end{center}
\vspace{1cm}

\centering \section*{Введение}

\begin{flushleft}
Реляционная СУБД хранит данные в виде таблиц (отношений), состоящих из строк и столбцов.
Для формулирования запросов к данным используется декларативный язык SQL.
С ростом объёма данных и сложности запросов увеличивается время, необходимое
для их выполнения, и сокращение этого времени становится важным фактором
эффективности СУБД.
\par

\textbf{Язык SQL.}
Стандартным средством взаимодействия с реляционными СУБД является язык SQL
(Structured Query Language)~--- декларативный язык запросов,
в котором пользователь описывает \emph{какие} данные необходимо получить,
но не указывает \emph{как} именно их извлечь.
Типичный запрос на чтение данных имеет вид
\texttt{SELECT~\ldots\ FROM~\ldots\ WHERE~\ldots}~и задаёт:
список возвращаемых атрибутов (проекция),
набор таблиц-источников и условия соединения между ними,
а также фильтрующие условия на строки.
Один и тот же результат может быть получен множеством различных
способов исполнения; выбор конкретного способа~--- задача оптимизатора СУБД.
\par

В процессе трансляции запрос превращается сначала в логическое представление в виде дерева, 
затем с помощью оптимизатора СУБД в физический план исполнения. Производительность СУБД напрямую зависит от качества
построенного физического плана. Для создания "хорошего" плана нужно решить или приблизить решение
NP-полной задачи выбора оптимального порядка соединений таблиц, так как оно требует сложных вычислений и значительных затрат ресурсов операционной системы.
В процессе работы оптимизатора, решается вопрос какой тип соединения использовать, как и в каком порядке соединить таблицы. Запрос преобразуется в набор планов.
\par

\textbf{Соединение таблиц} -- это операция, которая позволяет объединять данные
из двух и более таблиц по определённому условию. Использование соединений необходимо,
когда информация распределена между несколькими таблицами. Соединение принимает два аргумента, 
назовём первый аргумент левым, второй правым. Каждый аргумент это таблица, либо результат соединения нескольких 
таблиц.
\par

\textbf{Отношение} -- либо единичная таблица, либо результат соединения нескольких таблиц.
\par

\textbf{Кардинальность} отношения $R$~--- это число строк в $R$, обозначается $|R|$.
Для базовой таблицы кардинальность известна из системного каталога.
Для промежуточного результата $R_i \Join R_j$ кардинальность неизвестна заранее
и оценивается оптимизатором на основе статистик; обозначим оценку $\widehat{|R_i \Join R_j|}$.
\par

\textbf{Селективность} предиката соединения $p$~--- это доля строк декартова
произведения $R_i \times R_j$, удовлетворяющих условию $p$:
\[
    \mathrm{sel}(p) \;=\; \frac{|R_i \Join_p R_j|}{|R_i| \cdot |R_j|}, \qquad 0 \leqslant \mathrm{sel}(p) \leqslant 1.
\]
Кардинальность результата соединения выражается через селективность:
$|R_i \Join_p R_j| = \mathrm{sel}(p) \cdot |R_i| \cdot |R_j|$.
Оптимизатор оценивает $\mathrm{sel}(p)$ по статистикам (гистограммам, списку
наиболее частых значений), предполагая независимость атрибутов.
При нарушении этого предположения ошибки оценок селективности
каскадируются мультипликативно по цепочке соединений.


\textbf{Виды соединений таблиц}:
\begin{enumerate}
  \item \textbf{Внутреннее соединение} - возвращает только те строки, для которых условие связи выполняется в обоих аргументах.
  \item \textbf{Внешнее соединение} - возвращает все строки из одного или обоих аргументов, дополняющее 
  отсутствующие совпадения пустыми значениями. Существует три вида внешних соединений.
  \begin{enumerate}
        \item \textbf{Левое внешнее соединение} - возвращает все строки из левого аргумента и совпадения из правого.
        \item \textbf{Правое внешнее соединение} - возвращает все строки из правого аргумента и совпадения из левого.
        \item \textbf{Полное внешнее соединение} - возвращает объединение всех строк из обоих аргументов.
  \end{enumerate}
  \item \textbf{Полусоединение} - возвращает строки из левого аргумента по существованию хотя бы одного совпадения в правом аргументе.
  \item \textbf{Антисоединение} - это операция, обратная полусоединению, которая возвращает строки из левого аргумента только при отсутствии совпадений в правом.
  \item \textbf{Декартово произведение} - соединяет каждую строку левого аргумента с каждой строкой правого. Выполняет соединение без условия.
\end{enumerate}

\textbf{Стадии обработки запроса.}
Путь от текстового SQL-запроса до получения результата в PostgreSQL
проходит через несколько последовательных стадий:
\begin{enumerate}
    \item \textbf{Лексический и синтаксический анализ.}
    Текст запроса разбирается на токены и преобразуется в дерево разбора~--- синтаксическое представление запроса,
    отражающее его грамматическую структуру.
    На этом этапе проверяется корректность синтаксиса,
    но семантика запроса ещё не анализируется.

    \item \textbf{Семантический анализ.}
    Дерево разбора обогащается информацией из системного каталога:
    разрешаются имена таблиц и атрибутов, проверяются типы данных,
    раскрываются псевдонимы и представления (views).
    Результат~--- дерево запроса (query tree), в котором все объекты
    идентифицированы однозначно.

    \item \textbf{Переписывание.}
    К дереву запроса применяются правила переписывания:
    раскрытие представлений, применение правил безопасности на уровне строк,
    преобразование подзапросов.
    Семантика запроса сохраняется, но его структура может существенно измениться.

    \item \textbf{Логическая оптимизация.}
    Переписанное дерево преобразуется в логическое представление:
    определяется набор базовых отношений, извлекаются предикаты соединения
    и фильтрующие условия, строятся классы эквивалентности атрибутов.
    На этом этапе формируется граф соединений $G = (\mathcal{R},\, \mathcal{E})$,
    который является входом для алгоритмов выбора порядка соединений.

    \item \textbf{Физическая оптимизация (планирование).}
    Для графа соединений выбирается порядок соединения отношений,
    для каждого соединения определяется физический алгоритм исполнения
    (вложенный цикл, хеш-соединение, соединение слиянием),
    а также способы доступа к данным (последовательное сканирование,
    сканирование по индексу).
    Результат~--- физический план исполнения, представляющий собой
    дерево операторов с конкретными алгоритмами в каждом узле.
    Именно на этой стадии решается задача выбора порядка соединений,
    которой посвящена данная работа.

    \item \textbf{Исполнение.}
    Физический план передаётся исполнителю, который обходит дерево
    операторов и формирует результат запроса.
\end{enumerate}

Таким образом, задача оптимизации порядка соединений возникает на стадии
физической оптимизации, после того как из исходного SQL-запроса извлечены
базовые отношения $\mathcal{R}$, предикаты соединения $\mathcal{P}$
и построен граф соединений~$G$.


Оптимизатор СУБД использует стоимостную модель, которая преобразует прогнозы использования различных ресурсов в одно число. 
\textbf{Стоимость}. 
Ключевые компоненты, которые учитываются, это:
\begin{enumerate}
\item Затраты на ввод-вывод (I/O Cost): Чтение/запись данных с диска.
\item Затраты на ЦПУ (CPU Cost): Обработка данных (сравнения, вычисления, агрегация).
\item Затраты на передачу данных (Network Cost): Передача данных между узлами (в распределённых системах).
\item Затраты на память (Memory Cost): Использование оперативной памяти для временных структур.
\item Затраты на конкурентность (Concurrency Cost): Влияние на блокировки и параллельный доступ.
\end{enumerate}

Каждый план можно представить в виде дерева, вершинами в котором являются отношения. Рёбра — условия соединения отношений.
Операция внутреннего соединения коммутативна и ассоциативна по результату:
$(R_1 \Join R_2) \Join R_3$ и $R_1 \Join (R_2 \Join R_3)$ возвращают
одно и то же множество строк. Однако \emph{стоимость вычисления} этих
выражений, как правило, различна:
$\mathrm{Cost}\bigl((R_1 \Join R_2) \Join R_3\bigr) \neq
\mathrm{Cost}\bigl(R_1 \Join (R_2 \Join R_3)\bigr)$,
поскольку кардинальности промежуточных результатов зависят от порядка.
Внешние соединения (outer join) не ассоциативны и не коммутативны
уже на уровне семантики: перестановка аргументов может изменить результат.
Это дополнительно ограничивает множество допустимых порядков соединений.


Выбор, в какой последовательности нужно соединить таблицы, является задачей выбора 
порядка соединений. Различные планы исполнения для одного и того же запроса возвращают 
одинаковый результат, но время и ресурсы (CPU, память, I/O обращения, возможно 
сетевые ресурсы), необходимые для выполнения запроса, сильно различаются. Поэтому 
выбор оптимального плана позволяет сократить время отклика, минимизировать 
потребление ресурсов и эффективно обрабатывать большие массивы данных, значительно 
повышая удобство работы пользователя.
\par

Комбинаторная природа задачи видна через простую оценку.
Если соединения выполнять бинарно, план задаётся полным двоичным деревом
с $n$ листьями (таблицами). Число различных форм таких деревьев~---
$C_{n-1}$, где $C_k$~--- число Каталана.
С учётом перестановок таблиц по листьям число различных планов
составляет $C_{n-1} \cdot n!$; асимптотически
$C_{n-1} \cdot n! \sim \dfrac{4^{n-1} \cdot n!}{n^{3/2}\sqrt{\pi}}$.
Даже для $n = 10$ это $\approx 1{,}7 \cdot 10^{10}$ планов. Это  экспоненциальный рост количества возможных планов.
\par

В работе запрос рассматривается как (гипер)граф соединений. Вершины — отношения (таблицы), рёбра — условия соединений. Рёбра, 
затрагивающие более двух таблиц -- гипер-рёбра. 
Ситуацию осложняют внешние соединения (\texttt{Outer join}): в общем случае они семантически не ассоциативны и не коммутативны, 
следовательно, ограничивают допустимые переупорядочивания.
\par

Можно заметить, что гиперграф запроса можно разбить на различные топологии: цепи, циклы, звёзды, плотные графы. Количество 
возможных планов у двух топологий (с обычными рёбрами и без внешних соединений) от одинакового числа таблиц может отличаться в несколько 
раз, при увеличении количества таблиц разница может вырасти до нескольких порядков.
\par

На практике индустриальные СУБД комбинируют подходы. Для малого числа таблиц применяют динамическое программирование (перебор 
подмножеств с запоминанием лучших частичных планов). При росте количества таблиц переключаются на эвристики: жадные 
стратегии, локальные улучшения, генетические алгоритмы и их простые сочетания. У такого подхода есть существенный недостаток: 
количество возможных планов коррелирует с числом таблиц, но линейная зависимость отсутствует.
\par

В данной работе предлагаются методы адаптивного планирования запросов, которые включают в себя:
\begin{enumerate}
  \item Различные эвристики для планирования запроса.
  \item Критерии для выбора эвристик для планирования запроса.
  \item Критерии для динамического переключения между эвристиками во время планирования.
  \item Реализацию динамическое переключение между эвристиками в зависимости от критериев. 
\end{enumerate}

\textbf{Сквозной пример.}
Рассмотрим SQL-запрос, который находит названия фильмов жанра <<комедия>>,
вышедших после 2010 года, вместе с именами актёров:

\begin{center}
\begin{minipage}{0.85\textwidth}
\begin{verbatim}
SELECT t.title, n.name
FROM   title       AS t,
       cast_info   AS ci,
       name        AS n,
       movie_keyword AS mk,
       keyword     AS k
WHERE  ci.movie_id    = t.id
  AND  ci.person_id   = n.id
  AND  mk.movie_id    = t.id
  AND  mk.keyword_id  = k.id
  AND  k.keyword      = 'comedy'
  AND  t.production_year > 2010;
\end{verbatim}
\end{minipage}
\end{center}

После стадий разбора, семантического анализа и логической оптимизации
планировщик извлекает пять базовых отношений
$\mathcal{R} = \{t,\, ci,\, n,\, mk,\, k\}$
и четыре предиката соединения, образующие граф соединений~$G$:

\begin{center}
\begin{tikzpicture}[
    every node/.style={draw, circle, minimum size=1.1cm, font=\small},
    edge label/.style={draw=none, rectangle, font=\footnotesize, midway},
    node distance=2.8cm
]
    \node (t)  {$t$};
    \node (ci) [above right of=t]  {$ci$};
    \node (n)  [right of=ci] {$n$};
    \node (mk) [below right of=t]  {$mk$};
    \node (k)  [right of=mk] {$k$};

    \draw (t)  -- node[edge label, above left]  {$p_1$} (ci);
    \draw (ci) -- node[edge label, above]        {$p_2$} (n);
    \draw (t)  -- node[edge label, below left]  {$p_3$} (mk);
    \draw (mk) -- node[edge label, below]        {$p_4$} (k);
\end{tikzpicture}
\end{center}

\noindent
где $p_1\colon ci.\mathit{movie\_id} = t.\mathit{id}$,\;
$p_2\colon ci.\mathit{person\_id} = n.\mathit{id}$,\;
$p_3\colon mk.\mathit{movie\_id} = t.\mathit{id}$,\;
$p_4\colon mk.\mathit{keyword\_id} = k.\mathit{id}$.
Фильтрующие условия $k.\mathit{keyword} = \text{`comedy'}$ и
$t.\mathit{production\_year} > 2010$ применяются локально к таблицам $k$ и $t$.
Граф имеет топологию \emph{звезда с центром $t$} и двумя лучами:
$n - ci - t$ и $k - mk - t$.
\par

На стадии физической оптимизации планировщик перебирает порядки соединений.
Один из возможных планов~--- левосвязанное дерево:

\begin{center}
\begin{tikzpicture}[
    level distance=1.1cm,
    sibling distance=1.8cm,
    every node/.style={draw, circle, minimum size=0.7cm, font=\small},
    edge from parent/.style={draw, -}
]
\node {$\Join$}
    child {
        node {$\Join$}
        child {
            node {$\Join$}
            child {
                node {$\Join$}
                child { node {$k$} }
                child { node {$mk$} }
            }
            child { node {$t$} }
        }
        child { node {$ci$} }
    }
    child { node {$n$} };
\end{tikzpicture}
\end{center}
\begin{center}
$T_1 = (((k \Join mk) \Join t) \Join ci) \Join n$
\end{center}

Этот план начинает с малой таблицы $k$ (после фильтра $\mathit{keyword} = \text{`comedy'}$
остаётся одна строка), соединяет с $mk$ по индексу, затем присоединяет
$t$, $ci$ и $n$.
Промежуточные результаты растут постепенно, что минимизирует объём обработки.
Альтернативный порядок, в котором сначала соединяются большие таблицы
$ci$ и $t$ (кардинальности ${\sim}36\,\text{млн}$ и ${\sim}2{,}5\,\text{млн}$),
породил бы промежуточный результат в миллионы строк уже на первом шаге,
многократно увеличив время исполнения.
\par

Данный пример показывает, как структура графа соединений, селективность
предикатов и порядок соединений совместно определяют
эффективность плана исполнения.
